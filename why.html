the article

Your HTML fits in 14 KB. Your website is still slow.

There is a piece of web performance folklore that gets passed around every few months like it is breaking news: your website should be under 14 KB.

The reasoning is genuinely cool. When a browser opens a fresh TCP connection, the server is not allowed to blast bytes at full speed straight away. It has to start slow and ramp up, a thing called slow start. The first burst it can send before it has to stop and wait for an acknowledgement is roughly 10 packets, which works out to about 14 KB. So if your page fits in that first burst, the browser gets the whole thing in a single network round trip and can begin painting right away. Anything bigger pays for an extra round trip before the first pixel shows up.

It is a great fact. It is also, as a performance rule, almost completely useless. I know this because I built a tool to check, pointed it at 77 popular websites, and the numbers were not kind to the rule.

The setup

The 14 KB number describes the earliest moment the network will physically allow your page to paint. Call it the network floor. It is a real, honest lower bound. The question nobody seems to ask is whether anyone actually paints anywhere near it.

So I wrote a small CLI that computes the network floor straight from the bytes (no browser required, just protocol arithmetic) and then renders the same page in a real headless Chrome under a throttled mobile connection to measure when the first pixel actually lands. Divide the floor by the real paint time and you get a single number I called the Paint Readiness Ratio, or PRR.

A PRR of 1.00 means the page paints the instant the network allows. A PRR of 0.05 means the browser made the user stare at nothing for twenty times longer than the network required. Guess which one turned out to be more common.

The results

Across the 77 sites I could measure:

  • Mean network floor: about 608 ms.
  • Mean actual first paint: about 6,521 ms.
  • That is roughly a 10.7x gap between what the network allowed and what actually happened.
  • Correlation between document size and time to paint: approximately zero.

Let me restate that last one, because it is the whole point. The size of your HTML had basically no relationship with how fast the page painted. The 14 KB rule is a rule about a number that does not predict the thing everyone uses it to justify.

The fun part is the contradictions:

  • Spotify ships an HTML document that fits comfortably under 14 KB. It paints in about 13 seconds. It passes the rule and is one of the slowest things on the list.
  • Figma ships a document that sails past the budget by a wide margin. It paints in under a second.

Small and fast were never the same thing. The byte budget passed in both directions and told you nothing useful either time.

So what actually decides paint

JavaScript. That is the entire answer.

The network was ready in half a second, and the browser then spent the next several seconds downloading, parsing, and executing scripts before it was willing to show a single pixel. The 14 KB rule optimizes the one part of the loading process that was rarely the bottleneck.

This is not a hot take that JavaScript is evil. It is just a measurement. If your framework renders the page on the client, your first paint waits on that framework, and no amount of shaving HTML bytes will move it.

A brief and humbling interlude

Before I got too smug, I ran the tool on my own website, theharshal30.github.io.

It fits in 14 KB. The HTML document is 5.40 KB gzipped, comfortably inside the first round trip. By the sacred rule, my site is flawless.

The tool gave it a PRR of 0.31 and a label of "Moderately delayed." The network floor was 300 ms. The actual first paint was 968 ms. The culprit was a single render blocking script, support.js, which it correctly identified as a React runtime that hides the page with display:none until it has finished rendering. In other words, my own homepage is held hostage by JavaScript while its tiny, rule abiding HTML sits there doing absolutely nothing.

The tool was polite about it. It still pointed out the obvious fix, which is to pre-render the thing to static HTML instead of asking every visitor to boot a rendering engine just to read some text. Have I done that yet? No. I wrote a blog post instead. We all have priorities.

The point

The 14 KB rule is a lovely piece of networking trivia and a terrible performance metric. Fitting your HTML in the first congestion window is necessary if you ever want to paint at the floor, but it is nowhere near sufficient, and on the modern web it is rarely the thing standing between your user and the first pixel.

If you actually want to know how close your site paints to what the network allows, measure the ratio, not the bytes:

npx lightsoutt your-site.com --fcp

It gives you the floor, the real paint time, the ratio, and a risk graded list of what to fix, including the difference between "safely add defer here" and "this script renders your page, do not touch it without pre-rendering first."

And if you want to reach the floor instead of just measuring how far you are from it, its companion boxthis streams a tiny, JavaScript free critical shell first so the page paints on arrival, then streams the rest. Measure with one, fix with the other.

Your HTML being under 14 KB is a fine thing. Just stop pretending it makes you fast.
